Android Pentest Methodology

A structured, phase-based approach to testing Android applications — from initial reconnaissance through backend API validation. Based on OWASP MASVS / MASTG standards.
OWASP MASVSMASTGSTRUCTURED

Testing Phases Overview

Phase 1 — Reconnaissance

Gather APK, enumerate attack surface, identify app type, review AndroidManifest.xml, map components.

Phase 2 — Static Analysis

Decompile APK, find hardcoded secrets, insecure configs, certificate pinning implementation, exported components.

Phase 3 — Dynamic Analysis

Run the app with Frida/Objection attached. Monitor file writes, logs, memory, crypto operations, component interactions.

Phase 4 — Network Testing

Intercept HTTP/HTTPS traffic via Burp. Test API endpoints for auth issues, IDOR, injection, business logic flaws.

Phase 1 — Reconnaissance Checklist

  • Obtain APK from device (adb pull), APKPure, or APKCombo
  • Identify target package name: adb shell pm list packages | grep target
  • Review AndroidManifest.xml — exported components, permissions, intent filters
  • Identify app type: Native Java/Kotlin, React Native, Flutter, Cordova
  • Check minimum SDK and target SDK versions for version-specific vulnerabilities
  • Map all declared permissions — flag DANGEROUS and custom permissions
  • Record all exported Activities, Services, Receivers, and ContentProviders

Phase 2 — Static Analysis Checklist

  • Decompile with apktool d target.apk (Smali) and jadx-gui target.apk (Java)
  • Search for hardcoded secrets: API keys, tokens, passwords, private keys
  • Check Network Security Config (network_security_config.xml) for trust-anchors misconfig
  • Find all HTTP URLs — flag any http:// (cleartext traffic)
  • Check for allowBackup="true" on sensitive apps
  • Scan with MobSF for automated initial findings
  • Review custom SSL pinning implementation in code
  • Check ProGuard / R8 obfuscation level
  • Look for debug flags: android:debuggable="true"

Phase 3 — Dynamic Analysis Checklist

  • Set up Frida server on rooted device, attach with Objection
  • Run env in Objection to map all app directories
  • Check SharedPreferences: android sharedpreferences get
  • Inspect SQLite databases after key user flows (login, transactions)
  • Monitor logcat for sensitive data leakage during all user actions
  • Test all exported components with adb/Drozer (intent injection)
  • Attempt app backup: adb backup -apk com.target.app
  • Dump memory with fridump3 after login — search for tokens/keys
  • Monitor filesystem: list files before and after sensitive operations
  • Test biometric implementation — crypto-bound or boolean?
  • Check for clipboard leakage, screenshot exposure
  • Run root/emulator detection → bypass with Objection/Magisk/Frida

Phase 4 — Network Testing Checklist

  • Configure Burp Suite, install CA cert (adb push → /system/etc/security/cacerts)
  • Bypass SSL pinning (Objection android sslpinning disable or Frida script)
  • Map all API endpoints from proxied traffic
  • Test for IDOR — increment/swap object IDs across accounts
  • Decode and analyze JWT tokens — test alg:none, weak secrets
  • Check authorization on every endpoint — test with a lower-privilege token
  • Test rate limiting on OTP, login, and password reset endpoints
  • Identify mobile-specific headers (X-Device-ID, etc.) — attempt spoofing
  • Test for mass assignment — add unexpected fields to POST bodies

Android API Versions

Understanding API levels is the foundation of Android pentesting. The API level determines what features, permissions, and protections are available — and what attack surface exists.
RECON API LEVEL SDK

What is an API Level?

Every version of Android is assigned a unique integer called an API Level. When you install an app, Android checks the app's minSdkVersion and targetSdkVersion in the manifest. As a pentester, these values tell you exactly which security controls are available or absent.

💡
Why It Matters Apps with a low minSdkVersion may still support legacy APIs that have known vulnerabilities. A high targetSdkVersion means modern mitigations are in place (e.g., scoped storage, cleartext traffic restrictions). Always check both.

Key API Levels & Security Implications

24–257.xNougatNetwork Security Config. Cleartext traffic blocked. User CA certs no longer trusted by default for apps targeting 24+.
26–278.xOreoBackground execution limits. Autofill API. Instant Apps.

How to Extract API Info During a Pentest

From the Device

    

      

      adb shell
      
    

    # Get device API level
adb shell getprop ro.build.version.sdk

# Get full Android version
adb shell getprop ro.build.version.release

# Get security patch level
adb shell getprop ro.build.version.security_patch

# Check target app's SDK versions
adb shell dumpsys package com.target.app | grep -E "targetSdk|minSdk"

From the Manifest (after decompile)

    

      

      xml — AndroidManifest.xml
      
    

    <uses-sdk
    android:minSdkVersion="21"
    android:targetSdkVersion="33" />
⚠️
Attack Surface Alert If minSdkVersion is below 24, the app trusts user-installed CA certificates — meaning you can intercept HTTPS traffic without SSL pinning bypass! This is a huge win in older apps.

Pentest Checklist for API Levels

  • Is minSdkVersion below 17? → WebView JavaScript injection may be possible
  • Is targetSdkVersion below 24? → User CA certs trusted → Easy MITM
  • Is targetSdkVersion below 28? → Cleartext (HTTP) might be allowed
  • Is targetSdkVersion below 29? → Broad external storage read/write possible
  • Is targetSdkVersion below 31? → Activities can be trampolined (privilege escalation)
App Discovery · Topic 02

App Architecture

Understanding how Android apps are structured — from the APK container to the runtime environment — is essential before touching any tool.

The APK File

An APK (Android Package Kit) is simply a ZIP archive. Rename any .apk to .zip and extract it. You'll find:

AndroidManifest.xml

The app's blueprint — declares components, permissions, intent filters, min/target SDK. Encoded in binary XML inside APK, decoded with apktool.

classes.dex

Compiled Dalvik bytecode. Contains the app's logic. Multiple DEX files (classes2.dex, etc.) exist for large apps due to the 64K method limit.

resources.arsc

Compiled resource table — maps resource IDs to actual values (strings, layouts, drawables). Can contain hardcoded strings/URLs.

res/ & assets/

Raw resources: layouts, images, config files. assets/ is often where developers hide config JSON, certificates, or SQLite DBs.

lib/

Native shared libraries (.so files) compiled for different ABIs: arm64-v8a, armeabi-v7a, x86, x86_64. Can contain sensitive C/C++ logic.

META-INF/

APK signature files (MANIFEST.MF, CERT.SF, CERT.RSA). Used to verify integrity. Check certificate details here.

The Android Runtime (ART vs Dalvik)

Before API 21 (Android 5.0), apps ran on Dalvik — a register-based VM that ran DEX bytecode. From API 21, Android switched to ART (Android Runtime).

FeatureDalvikART
CompilationJIT (Just-in-Time)AOT (Ahead-of-Time) + JIT
File Format.dex.oat (compiled .dex)
PerformanceSlower startupFaster execution
Pentest ImpactEasier to hook with old toolsFrida/Objection needed for hooks

App Layers

01
Java/Kotlin Code
The main application logic, compiled to DEX bytecode. Decompilable to readable Java via JADX. This is where most vulnerabilities live.
02
Android Framework APIs
System-level APIs (Camera, Location, Storage, Network). App communicates through these. Misuse = data leaks.
03
Native Layer (C/C++ via JNI)
Performance-sensitive code or obfuscated logic moved to .so files. Harder to reverse-engineer but reversible with Ghidra or IDA Pro.
04
Linux Kernel
Android runs on a modified Linux kernel. Each app gets its own UID. The kernel enforces sandboxing via UIDs and SELinux policies.
ℹ️
Sandbox Model Each Android app runs in its own sandbox with a unique Linux user ID (UID). Apps cannot directly access each other's data without explicit IPC mechanisms (Intents, Content Providers, etc.). As a pentester, you're looking for misconfigurations in these IPC boundaries.

Android Debug Bridge (ADB)

ADB is your Swiss Army knife for Android pentesting. It's a command-line tool that lets you communicate with a connected Android device — think of it as SSH for Android.
ESSENTIAL RECON DATA EXTRACTION

Setup & Connection

    

      

      bash
      
    

    # Enable USB Debugging on device:
# Settings → About Phone → tap Build Number 7x → Developer Options → USB Debugging ON

# Start ADB server
adb start-server

# List connected devices
adb devices

# Connect over WiFi (same network)
adb tcpip 5555
adb connect <device_ip>:5555

# Open interactive shell
adb shell

# Shell as specific app user (requires root)
adb shell run-as com.target.app

App Discovery · Topic 04

App Permissions

Permissions define what an app can access. Over-privileged apps are a major attack surface. As a pentester, you map permissions to understand what data the app can leak.

Permission Categories

Normal Permissions

Automatically granted — low risk to user privacy. Example: INTERNET, SET_ALARM, VIBRATE

Dangerous Permissions

User must explicitly approve at runtime (API 23+). Involve sensitive data or hardware. Examples below.

Signature Permissions

Only granted to apps signed with the same certificate as the app that declared the permission. Used for IPC between apps from the same vendor.

Privileged Permissions

Only granted to apps installed in /system/priv-app/. Reserved for OEM/carrier apps.

Critical Dangerous Permissions to Look For

PermissionGroupPentest Relevance
READ_CONTACTSContactsPII exfiltration of full contact list
READ_CALL_LOGPhoneCall history — behavioral profiling
READ_SMSSMSOTP interception if app reads SMS
RECEIVE_SMSSMSIntercept incoming SMS (OTP theft)
ACCESS_FINE_LOCATIONLocationGPS tracking — high-risk PII
CAMERACameraPotential for unauthorized photo capture
RECORD_AUDIOMicrophoneEavesdropping risk
READ_EXTERNAL_STORAGEStorageRead all files on SD card
WRITE_EXTERNAL_STORAGEStorageWrite/overwrite external files
USE_BIOMETRICBiometricsFingerprint/face auth — test bypass
BIND_ACCESSIBILITY_SERVICEAccessibilityCan read screen content — keylogger risk

Extracting Permissions

    

      

      bash
      
    

    # List all permissions of a package
adb shell dumpsys package com.target.app | grep "permission"

# Check granted runtime permissions
adb shell dumpsys package com.target.app | grep "grantedPermissions" -A 50

# From decompiled manifest (aapt)
aapt dump permissions target.apk

# Using apktool after decode
grep -i "uses-permission" AndroidManifest.xml

Custom Permissions — An Often Missed Finding

Apps can declare their own custom permissions that other apps must hold to interact with them. If the protection level is misconfigured, any app can access the component.

    

      

      xml
      
    

    <!-- BAD: protectionLevel="normal" means ANY app can obtain this permission -->
<permission android:name="com.target.ACCESS_DATA"
    android:protectionLevel="normal" />

<!-- GOOD: Only apps with same signing cert get this permission -->
<permission android:name="com.target.ACCESS_DATA"
    android:protectionLevel="signature" />
🔴
High Severity FindingCustom permissions with protectionLevel="normal" allow any malicious app to declare the permission and access the protected component — a privilege escalation vulnerability.
App Discovery · Topic 05

Android App Types

The type of app determines your attack approach. Native apps, hybrid apps, and web apps each expose different attack surfaces.

Native Apps

Written in Java or Kotlin, compiled to DEX bytecode. Full access to Android SDK and device hardware. Most enterprise apps are native.

💡
Pentest ApproachDecompile DEX → JADX. Look for hardcoded keys, insecure IPC, SQL injection in ContentProviders, improper crypto usage.

Hybrid Apps

Use frameworks like React Native, Ionic, Cordova, Flutter, Xamarin. The UI runs in a WebView or canvas, with native bridges for device APIs.

FrameworkLanguageKey Files to Look At
React NativeJavaScriptassets/index.android.bundle — readable JS bundle! Full source code.
FlutterDartlibapp.so — compiled Dart kernel. Use reFlutter or blutter to extract.
Cordova/IonicHTML/JS/CSSassets/www/ — full HTML/JS source code in plaintext!
XamarinC#assemblies/*.dll — .NET assemblies, decompile with dnSpy or dotPeek.
⚠️
React Native is a Gold MineThe JavaScript bundle is rarely obfuscated. You can find API keys, secrets, and full business logic by simply reading index.android.bundle with any text editor or beautifier.

WebView Apps

Apps that load a website inside a WebView component. The entire app is a browser. Look for:

  • JavaScript Interface exposure — addJavascriptInterface() allows JS to call native Java methods
  • Universal File Access — setAllowFileAccessFromFileURLs(true) allows XSS to read local files
  • Insecure WebView settings — debug mode enabled, cleartext loading
  • Deep link → WebView injection — URL passed to WebView without validation
    

      

      java — Dangerous WebView Config
      
    

    // 🚨 DANGEROUS: exposes native Java to JavaScript
webView.addJavascriptInterface(new JavaScriptBridge(), "Android");

// 🚨 DANGEROUS: allows file:// XSS
webView.getSettings().setAllowFileAccessFromFileURLs(true);
webView.getSettings().setAllowUniversalAccessFromFileURLs(true);

// 🚨 DANGEROUS: loads attacker-controlled URL from Intent
String url = getIntent().getStringExtra("url");
webView.loadUrl(url); // No validation!
App Discovery · Topic 06

App Components

Android apps are built from four fundamental building blocks. Each one has its own attack surface. Mastering them is critical for finding IPC vulnerabilities.

1. Activity

A single screen with a UI. The entry point for user interaction. Activities can be exported — accessible from other apps via Intents.

    

      

      xml — AndroidManifest.xml
      
    

    <!-- 🚨 Exported without permission = any app can launch it -->
<activity android:name=".AdminActivity"
          android:exported="true" />

<!-- Testing: Launch exported activity via ADB -->
# adb shell am start -n com.target/.AdminActivity
🔴
Auth BypassExported Activities without permission checks can be launched directly, bypassing login screens. Always test every exported activity!

2. Service

Runs background operations (no UI). Can be Started (fire-and-forget) or Bound (IPC via IBinder). Exported services are callable from other apps.

    

      

      bash
      
    

    # Test exported service via ADB
adb shell am startservice -n com.target.app/.SyncService \
  --es "action" "dump_data"

# Check exported services
adb shell dumpsys package com.target.app | grep -A2 "Service"

3. Broadcast Receiver

Listens for system-wide or app-specific broadcast events (e.g., SMS received, boot completed). If exported and unprotected, any app can send malicious broadcasts.

    

      

      bash
      
    

    # Send a broadcast to test exported receiver
adb shell am broadcast \
  -a com.target.app.SECRET_ACTION \
  --es "command" "delete_all_data"

# List registered receivers from manifest
grep -i "receiver" AndroidManifest.xml

4. Content Provider

Manages shared app data via a URI interface (like a mini REST API). Exposes query, insert, update, delete. Misconfigured providers are a goldmine for data theft and SQL injection.

    

      

      bash
      
    

    # Query a Content Provider directly
adb shell content query \
  --uri content://com.target.app.provider/users

# SQL Injection via Content Provider
adb shell content query \
  --uri content://com.target.app.provider/users \
  --where "1=1 OR 1=1--"

# Using Drozer to enumerate providers
dz> run app.provider.finduri com.target.app
dz> run app.provider.query content://com.target.app.provider/users
⚠️
Path Traversal in File ProvidersFileProvider exposing files with android:grantUriPermissions="true" without proper path restrictions can lead to arbitrary file read. Test with content://provider/../../../etc/hosts-style payloads.

Static Analysis · Topic 01

Decode & Decompile APK File

Static analysis starts with getting the APK and ripping it open. You'll use different tools depending on what you need — decoded resources, readable Java code, or raw bytecode.

Step 1 — Get the APK

    

      

      bash
      
    

    # Method 1: Pull from device
adb shell pm path com.target.app
adb pull /data/app/~~xxx==/com.target.app-1/base.apk

# Method 2: APKPure / APKMirror (for public apps)
# Method 3: Using gplaycli (Google Play CLI)
pip install gplaycli
gplaycli -d com.target.app -f ./

# Method 4: Backup trick (no root needed)
adb backup -noapk com.target.app
dd if=backup.ab bs=1 skip=24 | python -c \
  "import zlib,sys; sys.stdout.buffer.write(zlib.decompress(sys.stdin.buffer.read()))" \
  | tar -xvf -

Step 2 — Decode with apktool (Resources + Manifest)

    

      

      bash
      
    

    # Decode APK (resources + binary manifest → readable)
apktool d target.apk -o decoded/

# Output structure:
# decoded/AndroidManifest.xml   ← decoded XML
# decoded/res/                  ← string resources, layouts
# decoded/smali/                ← Dalvik bytecode in Smali ASM
# decoded/assets/               ← raw assets

# Repackage (for patching)
apktool b decoded/ -o patched.apk

Step 3 — Decompile to Java with JADX (Best for Reading Code)

    

      

      bash
      
    

    # GUI version (recommended)
jadx-gui target.apk

# CLI — decompile to Java source files
jadx -d output/ target.apk

# Search for patterns in decompiled output
grep -rn "password\|secret\|api_key\|token" output/

# JADX flags
jadx --show-bad-code target.apk   # Show even problematic decompilation
jadx --deobf target.apk           # Basic deobfuscation

Step 4 — dex2jar + JD-GUI (Alternative Java Decompiler)

    

      

      bash
      
    

    # Convert DEX to JAR
d2j-dex2jar.sh target.apk -o target.jar

# Open in JD-GUI
jd-gui target.jar

# Or use CFR decompiler (better for lambdas/streams)
java -jar cfr.jar target.jar --outputdir ./cfr_output/

Step 5 — Signing a Modified APK

    

      

      bash
      
    

    # Generate a test key
keytool -genkey -v -keystore test.keystore \
  -alias test -keyalg RSA -keysize 2048 -validity 10000

# Sign the patched APK
jarsigner -verbose -sigalg SHA1withRSA -digestalg SHA1 \
  -keystore test.keystore patched.apk test

# Or use apksigner (preferred)
apksigner sign --ks test.keystore patched.apk

# Align the APK
zipalign -v 4 patched.apk patched-aligned.apk

# Install
adb install patched-aligned.apk
Static Analysis · Topic 02

Binary Protections

Modern apps use protections to resist reverse engineering. Your job is to identify which protections are in place, assess their effectiveness, and find ways around them.

Types of Binary Protections

1. Code Obfuscation (ProGuard / R8 / DexGuard)

ProGuard/R8 rename classes, methods, and fields to meaningless single-character names (a, b, c...). DexGuard (commercial) adds string encryption, class encryption, and anti-tamper.

    

      

      java — Obfuscated vs Clear
      
    

    // BEFORE obfuscation
class LoginManager {
    String apiKey = "sk_live_abc123";
}

// AFTER ProGuard
class a {
    String a = "sk_live_abc123"; // Strings NOT encrypted by default!
}
💡
Analyst TipProGuard obfuscates names but NOT string values. Hardcoded API keys, URLs, and secrets remain readable even in obfuscated code!

2. Root Detection

Apps check for signs of a rooted device: presence of su binary, Magisk, SuperSU, test-keys build, etc. Covered in the Bypasses section.

3. Emulator Detection

Apps detect if they're running on an emulator by checking build properties, IMEI, sensor data, etc.

    

      

      java — Emulator Detection Checks
      
    

    Build.FINGERPRINT.startsWith("generic")
Build.MODEL.contains("google_sdk")
Build.MANUFACTURER.contains("Genymotion")
Build.BRAND.startsWith("generic")
Build.PRODUCT.equals("sdk")

Bypass: Use a real device, or patch the APK with apktool to change these checks, or hook with Frida.

4. Anti-Tamper / Signature Checking

Biometric Testing

Biometric authentication (fingerprint, face) is increasingly common. Implementations vary widely in security. The goal is to determine if the biometric check actually protects the underlying secret or just gates access.

Two Implementation Types

✅ Crypto-Bound (Secure)

A structured, phase-based approach to testing iOS applications — from IPA acquisition and decryption through runtime analysis and network interception. Aligned with OWASP MASVS / MASTG standards.

Testing Phases Overview

Phase 1 — Acquisition

Obtain IPA, decrypt FairPlay DRM (jailbroken device), identify app type, inspect Info.plist and provisioning profile.

  • Start frida-server on device, attach: objection -g com.target.app explore

App Discovery · Topic 02

App Environment

Every iOS app lives in a strictly enforced sandbox. Understanding this environment — the filesystem layout, data container model, and runtime protections — is the foundation of iOS pentesting.

iOS App Sandbox

Each iOS app is sandboxed with a unique UUID-based directory. Unlike Android, apps cannot interact with the broader filesystem. There are two main containers per app:

Bundle Container

Read-only. Contains the app binary, resources, Info.plist, frameworks. Path: /var/containers/Bundle/Application/<UUID>/

Data Container

Read-write. Contains app-generated data. Path: /var/mobile/Containers/Data/Application/<UUID>/

The iOS Security Model

  • Mandatory Access Control (MAC) — kernel-enforced via sandboxd. Sandbox profiles (SBPL) define what each app can and cannot do.
  • Code Signing — every binary must be signed by Apple or a trusted certificate. Jailbreaks bypass this for security testing.
  • Data Protection — files can be encrypted with keys derived from the device passcode + hardware key. Class-based (covered separately).
  • ASLR + PIE — address space layout randomization. Makes exploit offsets unpredictable.
  • No Root User — apps run as mobile user (uid 501). Root is only accessible post-jailbreak.

When you download an app from the App Store, the binary is encrypted. The decryption key is tied to your Apple ID and device hardware. The iOS kernel decrypts it into memory when launched — meaning you can dump the decrypted binary from memory at runtime.

Static Analysis · Topic 02

Binary Protections

iOS binaries should have multiple hardening protections enabled. Checking these is a required step in any iOS pentest — missing protections are a finding in themselves.

Protection Checklist with checksec / otool

    
bash

    # Using otool to check all protections

── 1. PIE (Position Independent Executable) ──
otool -hv TargetApp | grep PIE
# Should see: PIE flag set
# Missing PIE = ASLR not effective → fixed memory addresses = easier exploitation

── 2. Stack Canary ──
otool -I -v TargetApp | grep stack_chk
# Should see: ___stack_chk_guard, ___stack_chk_fail
# Missing = stack buffer overflows undetected

── 3. ARC (Automatic Reference Counting) ──
otool -I -v TargetApp | grep objc_release
# ARC protects against use-after-free vulnerabilities

── 4. FairPlay Encryption (DRM) ──
otool -l TargetApp | grep -A4 LC_ENCRYPTION_INFO
# cryptid = 1: encrypted (good for distribution, bad for analysis)

── 5. Rpath ──
otool -l TargetApp | grep -A3 LC_RPATH
# Insecure rpath = dylib hijacking possible

── 6. Symbols Strip ──
nm TargetApp | wc -l
# Many symbols = debug build not stripped → easier to reverse

Using checksec (All at Once)

    
bash

    # Install
pip3 install checksec.py

# Run against binary
checksec --file=TargetApp_arm64

# Or for all binaries in the app
find extracted/Payload/ -type f -exec file {} \; | grep Mach-O | \
  awk -F: '{print $1}' | xargs -I{} checksec --file={}

Transport Layer Security

Verifying TLS is properly configured is the first dynamic test. ATS (App Transport Security) is iOS's built-in enforcement mechanism — but it can be disabled.

ATS Testing Checklist

  • Is NSAllowsArbitraryLoads set to true? → HTTP to any domain allowed
  • Are there NSExceptionDomains with NSExceptionAllowsInsecureHTTPLoads?
  • Is TLS version downgraded (NSExceptionMinimumTLSVersion: TLSv1.0)?
  • Is NSAllowsArbitraryLoadsInWebContent true? → WebView can load HTTP
  • Is NSAllowsLocalNetworking true? → Local network HTTP allowed

Test TLS Version and Cipher Suites

    
bash

    # Test server TLS configuration

# Check minimum TLS version accepted
openssl s_client -connect api.target.com:443 -tls1
openssl s_client -connect api.target.com:443 -tls1_1
# If these succeed → server accepts weak TLS

# Test with testssl.sh (comprehensive)
testssl.sh api.target.com

# Verify certificate validity
openssl s_client -connect api.target.com:443 2>/dev/null | \
  openssl x509 -noout -dates -subject -issuer

Biometrics Testing

Touch ID and Face ID use LocalAuthentication framework. The security depends entirely on whether biometric success is cryptographically bound to Keychain access or just returns a boolean — the latter is trivially bypassable.

Two Implementation Patterns

✅ Secure: Keychain-Bound

Biometric auth unlocks a Keychain item with kSecAccessControlBiometryCurrentSet. The secret never leaves Secure Enclave without real biometric auth. Frida can't bypass this.

App calls evaluatePolicy, waits for true, then proceeds. Frida hooks the callback and returns true without any real authentication. Trivially bypassed.

Using FLEXible to Record HTTP Communications

FLEX is a powerful in-app debugging library that can be injected into any iOS app on a jailbroken device. It provides an HTTP request history, runtime inspector, and file browser — entirely on-device with no proxy needed.

What is FLEX?

FLEX (Flipboard Explorer) is an in-app debugging tool that shows: network request history, response bodies, headers, view hierarchy, NSUserDefaults, Keychain items, and runtime class inspection — all in an overlay UI inside the target app.

Using FLEX for HTTP Monitoring

01
Shake device or tap toolbar icon
The FLEX toolbar appears at the bottom of the screen inside the app.
02
Tap "Network History"
Shows all HTTP/HTTPS requests made by the app — even requests to pinned endpoints that Burp can't see!
03
Tap any request
See full request headers, body, response headers, response body, timing, and status code.
04
Browse Runtime Data
Tap "Globals" → explore NSUserDefaults, Keychain, live object graph, file system.

Why FLEXible is Powerful for Pentesting

Bypasses Proxy Detection

Since it hooks at the URLSession level inside the process, it sees all requests — including those that explicitly avoid the system proxy.

Bypasses SSL Pinning (Indirectly)

The network history shows both request and response body. Even if the app pins certificates, FLEX sees the decrypted data because it's hooked inside the app process itself.

NSUserDefaults Inspector

Live view of all NSUserDefaults values. Can modify values in real-time to test privilege escalation without writing Frida scripts.

Filesystem Browser

Full in-app file browser of the app's sandbox. View files, plist content, and SQLite databases without SSH.

💡
FLEX + SSL PinningFLEX is especially useful when you can't bypass SSL pinning with other tools — it shows you the decrypted API request and response from inside the app process, letting you document the API surface without MITM.

Inject FLEX Without Jailbreak (Patched IPA)

    
bash

    # Inject FLEX framework into IPA manually

# 1. Unzip IPA
unzip target.ipa -d extracted/

# 2. Copy FLEX.framework into Frameworks/
cp -r FLEX.framework extracted/Payload/TargetApp.app/Frameworks/

# 3. Add load command to binary (insert_dylib or optool)
optool install -c load -p "@rpath/FLEX.framework/FLEX" \
  -t extracted/Payload/TargetApp.app/TargetApp

# 4. Add FLEX initializer to binary (via custom dylib or patch)
# 5. Re-sign and install
codesign -f -s "iPhone Developer: ..." \
  extracted/Payload/TargetApp.app/Frameworks/FLEX.framework
codesign -f -s "iPhone Developer: ..." \
  extracted/Payload/TargetApp.app/TargetApp
zip -r patched.ipa extracted/Payload/
ideviceinstaller -i patched.ipa
